# Restore Newcomer Guardrails

## Plan

**Goal:** Make Superpowers safe by default for users who do not know workflow names, engineering jargon, or which checks to request, without restoring universal durable artifacts.
**Why planning is required:** This corrects the router's public lifecycle contract and must keep runtime metadata, validation, and bilingual guidance consistent.
**Acceptance:** Only provably non-behavioral mechanical edits use the direct path; every other implementation follows an explicit lightweight inspect-plan-implement-verify-review loop; loading a workflow never grants mutation authority; outcome-based metadata selects specialist skills without requiring user jargon; High-risk adds to Standard and retains durable planning, independent verification, and conditional production-data/deployment safeguards; validation passes within the existing context budget.

### Outcome 1: Enforce a mandatory core implementation loop
- Work: Narrow the direct path to provably non-behavioral changes without data, dependency, security, public-contract, or external-state impact. Require a brief plan, practical focused coverage, strongest focused checks, final diff/acceptance review, and evidence report for every other implementation.
- Verify: forward-test vague newcomer prompts for a typo and a routine behavior change.

### Outcome 2: Make risk escalation fail safe
- Work: Require upward classification under uncertainty and make High-risk cumulative with Standard. For security-sensitive, destructive, production-data, migration, public-contract, cross-system, or consequential external-state work, require durable planning and explicit verification; add focused review or isolation when their named risk applies.
- Verify: forward-test a vague destructive cross-module request and inspect authorization, recovery, review, and verification decisions.

### Outcome 3: Keep the bundle concise and internally consistent
- Work: Replace obsolete fast-path validator contracts; update English and Traditional Chinese guides, README positioning, and measured metrics only where behavior or counts change.
- Verify: `bash skills/superpowers/check-context-budget.sh`, changed-skill `quick_validate.py`, `bash -n`, local-link validation, and `git diff --check`.

### Outcome 4: Make TDD discoverable without making it universal
- Work: Trigger `test-driven-development` from a cheap, focused red test that materially reduces ambiguity or regression risk, not from the user's knowledge of TDD terminology. Retain explicit exclusions where test-first is infeasible, artificial, or disproportionate.
- Verify: forward-test a novice routine implementation prompt that never names TDD and a mechanical edit where TDD must remain inactive.

### Outcome 5: Deliver the supported correction
- Work: Record the verified result, reconcile plan/log/diff/evidence, commit only this scope, and push under repository policy.
- Verify: clean local/remote status at the pushed commit.

## Decision

- Restoring the universal plan/log/reconciliation chain would protect novices but recreates the measured artifact and tool-call overhead.
- Keeping routine work fully direct assumes users or agents will volunteer missing planning and review steps; the current router provides no such guarantee.
- Chosen balance: mandatory lightweight discipline inside the always-on router, with durable files and extra workflow skills reserved for material risk. User silence never waives the core loop.
- TDD follows the same principle: Codex selects it from observable task shape, while an explicit benefit threshold prevents a universal red-green ceremony.

## Novice architecture audit

| Blind spot | Decision |
| --- | --- |
| Behavior-preserving code, test-only, and configuration work could fall between Mechanical and Standard | Make Standard cover every non-mechanical implementation; High-risk explicitly includes Standard. |
| Debugging, TDD, review feedback, or an existing plan could be mistaken for edit authority | Add diagnosis/evaluation/plan-only stop rules and require implementation authorization before mutation. |
| TDD required the user to know its name, then became broad enough to match almost any testable change | Auto-select only for a clear expected behavior where a cheap red test disambiguates implementation or captures a confirmed regression. |
| Debugging fixed before adding a regression test and had no unreproducible-failure stop | Give debugging root-cause ownership, TDD red-green ordering when independently matched, and an evidence-insufficient fallback. |
| High-risk verification was both mandatory and skippable; routine multi-command checks could over-trigger it | Make router/user/repository-required gates non-skippable, reuse fresh evidence, and exclude routine command sets as a trigger. |
| Incoming and independent outgoing review could both activate | Encode direction in metadata and keep evaluation-only feedback read-only. |
| Initial routing could become stale after inspection | Reclassify only when new scope/risk appears and before consequential external action; reuse unchanged evidence. |
| Generic High-risk rules left production deployment and destructive data controls implicit | Add conditional target, rollout/rollback, dry-run, recovery, integrity, and post-operation evidence inside plan/gate skills; add no new framework. |
| Push failures and validator inventory replacement had unsafe fallbacks | Preserve verified state without implicit pull/rebase/force; validate exact skill identities and front matter. |
| English jargon might under-trigger for novice or multilingual prompts | Prefer semantic outcome triggers; do not add brittle language-specific keyword lists. |
